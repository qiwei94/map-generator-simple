# 桥梁过滤效果总结

## 过滤策略

根据 `manifold_boolean_spec.md` 第192-215行的桥梁处理策略：

### 核心逻辑

1. **筛选 bridge=yes 标签的道路**
   - OSM数据中显式标记的桥梁
   - 如无显式标记，则筛选所有与水体相交的道路

2. **计算道路与水体的交集**
   - 使用 Shapely 的 `intersection()` 方法
   - 只保留道路在水体内的片段（纯桥梁段）

3. **返回过滤结果**
   - 过滤率：约 99% 的道路被过滤掉
   - 只保留跨越水体的桥梁段

## 杭州西湖区域预期效果

### 输入数据（25km × 25km）

- **道路数**：约 3,000-5,000 条
- **水体数**：约 50-100 个（钱塘江、西湖等）
- **数据获取时间**：
  - 道路：30-60 秒（Overpass API）
  - 水体：10-20 秒

### 过滤结果

- **bridge=yes 标记**：约 10-30 条
- **与水体相交的道路**：约 50-100 条
- **过滤后桥梁**：约 5-20 条（纯桥梁段）

### 典型桥梁（杭州）

- 钱塘江大桥：钱江一桥、二桥、三桥、四桥
- 西湖周边的小桥
- 其他水体上的桥梁

## 优化建议

### 1. 使用缓存（已实现）

- 道路缓存：`F:/map_gen_cache/project_cache/osm/`
- 水体缓存：同上
- 缓存策略：优先模式，F盘优先，C盘降级

### 2. 先过滤再生成网格

```python
# 正确流程：
roads_gdf → filter_bridges_only() → build_deepseek_roads()

# 避免：
roads_gdf → build_deepseek_roads() → 后期过滤（浪费计算）
```

### 3. 批量布尔运算

- **水体镂空**：批量Union + 单次Subtract（已实现）
- **道路融合**：单次Union（桥梁数量少，开销小）
